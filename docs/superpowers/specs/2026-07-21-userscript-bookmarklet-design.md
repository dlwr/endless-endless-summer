# userscript / ブックマークレット版 設計

2026-07-21 ブレインストーミングで確定。

## 背景

Tumblr API のレートリミット緩和申請が却下された。現行の Web アプリ構成(単一 consumer key を全ユーザーで共有、1,000/hr・5,000/day)では公開サービスとしてスケールしない。そこで公開形態を「tumblr.com 上で動くユーザースクリプト/ブックマークレット」に転換する。ユーザー自身のログインセッション(内部 API)を使うため consumer key が不要になり、レートリミットはユーザー単位となって構造的に問題が消える。

2026-07-17 設計時にユーザースクリプト案を却下した理由は「キーボードショートカットの自前実装が必要」だったが、本設計では描画・操作を Tumblr 本体に任せるため、この理由は該当しない。

## 決定事項

| 論点 | 決定 |
|---|---|
| レートリミット回避 | Tumblr セッション相乗り(内部 API `www.tumblr.com/api/v2`) |
| 配布形態 | **ユーザースクリプト単独**(2026-07-21 spike でブックマークレットは fetch 介入方式と非互換と判明。下記) |
| 既存 Web アプリ | 自分専用として存続。コアロジックを共有パッケージに切り出す |
| UI | ダッシュボードへの注入(自前 UI は持たない) |
| 注入方式 | API レスポンス介入(fetch フック)。DOM には触らない |
| 置換度合い | 完全置換モード(タイムライン全体をランダム過去ポストにする)。オン/オフトグル付き |

> **2026-07-21 spike 結果**: 成立性を実機検証済み(`docs/superpowers/specs/2026-07-21-userscript-spike-report.md`)。captured token で内部 API 到達・fetch 介入・ネイティブ描画まで成立。ブックマークレットは撤回。以下の設計はその結果を反映済み。

## アーキテクチャ

モノレポに 2 パッケージを追加する。

- `packages/core/`(新規): 年均等ランダムサンプリング、`before` タイムスタンプ計算、最古ポスト日時キャッシュのロジックを Workers 版から切り出す。加えて **snake→camel 深変換** と **自前タイムライン封筒生成**(後述)を持つ。ストレージ(KV / GM ストレージ / localStorage)と API クライアント(公式 API / 内部 API)はインターフェースで抽象化し、各実装から注入する。既存の該当テストはここへ移動。
- `packages/userscript/`(新規): fetch フック本体、内部 API クライアント、captured token 管理、トグル UI、ユーザースクリプトのビルド。
- 既存 Workers + SPA: `core` を import する形にリファクタし、自分専用として維持。

## 動作フロー

1. `@run-at document_start` で Tumblr 本体より先に `window.fetch` をラップする。**これが成立の前提**: アプリは初期化時に `window.fetch` をキャッシュするため、先にラップすればアプリがキャッシュするのが我々のラッパーになる。ロード後起動では素通りする(spike 実証)。
2. ダッシュボードタイムライン API(`/api/v2/timeline/dashboard`)への送信リクエストのヘッダーから **Bearer トークンを捕獲**する(値はメモリ内に留め、保存も露出もしない)。cookie 単独では内部 API は 401、captured token で 200(spike 実証)。
3. 捕獲トークンで内部 API を叩く: フォロー一覧 → ランダムブログ選択 → `/api/v2/blog/{name}/posts?before=…` でランダム過去ポストを取得。**取得元は snake_case、注入先(dashboard elements)は camelCase** なので、`packages/core` の深変換で camelCase 化してから注入する(フィールド集合は同一)。
4. レスポンスの `timeline.elements` を(camel 化した)ランダム過去ポストに差し替えて返す。描画は Tumblr 本体の React が行うため、リブログ・ライク・ノート表示・本家キーボード操作はネイティブに動く(spike で描画・リブログ・`J/K/T/L` 実証済み)。
5. **ページングの一貫生成が必須**: 丸ごと固定置換は同一 ID の返却で無限ローディングに陥る(spike で確認)。`_links.next` を自前カーソルに差し替え、要素ごとに `streamGlobalPosition`/`streamSessionId` を単調生成し、ページごとに新しいランダムポスト・ユニーク ID を返して前進させる。
6. **初回画面の扱い**: 初回表示分はサーバー埋め込み(`___INITIAL_STATE___`)で fetch を経由しない可能性が高い。起動直後に再取得を誘発するか、埋め込み state を書き換えて初回から置換する。
7. キャッシュ(最古ポスト日時等)は `GM_setValue` があれば使い、なければ `localStorage`。
8. 1 ページ分に必要な件数しか内部 API を叩かない(ユーザー自身のセッションでも行儀よく)。

## 配布形態(ユーザースクリプト単独)

esbuild で `.user.js`(Tampermonkey ヘッダー付き、`@run-at document_start`、ページ内トグル + GM メニューでオン/オフ)を出力する。

**ブックマークレットは非対応**(2026-07-21 spike)。ブックマークレットはページロード後のクリック起動で、その時点でアプリは既に `window.fetch` をキャッシュ済みのため、後からラップしてもアプリのタイムライン取得を介入できない。fetch 介入方式と原理的に両立しない。実現するには DOM/React 直接注入という別方式が必要で、これは壊れやすく却下済み。

## エラー処理

フック内で何が起きても必ず素のレスポンスにフォールバックし、ダッシュボードを壊さないことを最優先とする。内部 API の形状変化でパースに失敗したら静かに置換をやめて console に警告を出す。

**follow-up(ハードニング)**: 現状のフックは `buildElements`(内部 API での donor 取得)を無タイムアウトで await する。donor 取得が**例外**なら passthrough で素のダッシュボードに戻る(対応済み)が、**ハング**(応答の遅い/無反応なブログ)した場合はそのスクロールページのスピナーが止まらない懸念がある。`AbortController` + タイムアウトで一定時間内に donor が揃わなければ passthrough する処理を追加する。

## テスト

- コアロジック: 既存の Vitest 資産を `packages/core/` へ移して維持。
- レスポンス置換: 実レスポンスをサニタイズした fixture JSON で TDD。
- 実機: Tampermonkey で確認。

## Spike(実施済み・2026-07-21)

`spike/05-spike.user.js` を段階実行して検証。詳細は `2026-07-21-userscript-spike-report.md`。

1. **内部 API 到達**(検証 A): 成立。cookie 単独 401 → captured token で following/posts とも 200。
2. **介入・描画**(検証 B): 成立。document_start の fetch ラップで dashboard レスポンスを書き換え、React がネイティブ再描画(反転・注入とも確認)。
3. **スキーマ差**: dashboard=camelCase / blog posts=snake_case、フィールド同一。深変換で橋渡し(実証)。
4. **丸ごと置換の失敗**: ページングが壊れ無限ローディング → 自前封筒生成が必要。
5. **ブックマークレット非互換**(旧検証 C): 成立せず。ロード後起動では fetch 介入不可 → 形態撤回。
6. **ネイティブキーボード**: 成立。注入ポストでも `J/K`=移動、`L`=ライク、`Shift+R`=リブログが動作。**`T` はネイティブではない**(実装 E2E で判明)ため、`t`=即リブログ含む独自スキームは userscript で自前実装が必要(follow-up)。GIF 自動再生は Tumblr/OS 設定依存で本方式とは無関係。
